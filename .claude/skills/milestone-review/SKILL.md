# /milestone-review — 마일스톤 커밋 전 파급력 검토

마일스톤 작업 완료 후 커밋 전, 현재 구현이 이후 마일스톤에 파급력 있는 문제를 포함하는지 검토한다.

## 사용법

```
/milestone-review M1
```

인자로 현재 완료한 마일스톤 이름(예: `M1`, `M2`, …)을 받는다.

---

## 검토 원칙

- **지금 안 고치면 이후 마일스톤에서 큰 비용이 드는 문제만** 리포트한다
- 나중에 추가/수정해도 파급 없는 항목은 "나중에 해도 됨"으로 분류한다
- 코드 스타일, 주석, 네이밍 등 사소한 사항은 리포트하지 않는다

---

## 실행 절차

아래 절차를 순서대로 수행하라:

1. **인자 파싱**: 첫 번째 인자를 마일스톤 이름으로 사용 (예: `M1`)
   - 인자가 없으면 사용자에게 마일스톤 이름을 되물어라
2. **milestones.md 읽기**: `docs/milestones.md`를 읽어 해당 마일스톤의 존재 여부, 의존 관계, 이후 마일스톤 목록을 파악
   - 마일스톤 이름이 `docs/milestones.md`에 없으면 사용자에게 경고 후 마일스톤 이름을 되물어라
3. **스펙 읽기**: `docs/system-design.md`를 읽어 전체 시스템 스펙 파악
4. **변경 파일 수집**: git diff/log로 현재 마일스톤에서 변경/생성된 파일 목록을 수집하고 해당 파일들의 코드를 읽기
5. **이후 마일스톤 요구사항 수집**: 현재 마일스톤에 의존하는 이후 마일스톤들의 세부 문서(`docs/milestones/*.md`)를 읽기
   - 세부 문서가 없는 마일스톤이 있으면 경고를 출력하되, 있는 정보만으로 검토를 계속한다
6. **대조 분석**: 현재 구현이 이후 마일스톤의 요구사항과 충돌하거나, 누락으로 인해 나중에 큰 수정이 필요한 부분을 식별
7. **리포트 작성**: 아래 포맷으로 결과를 출력

---

## 리포트 포맷

```markdown
## 검토 결과: {마일스톤}

### 지금 수정 필요 (파급력 높음)
- [{파일}:{라인}] {문제 설명} — 영향받는 마일스톤: {M?}

### 나중에 해도 됨
- {항목} — {이유}

### 결론
{커밋 가능 여부 한 줄 요약}
```

문제가 전혀 없으면:

```markdown
## 검토 결과: {마일스톤}

### 지금 수정 필요 (파급력 높음)
없음

### 나중에 해도 됨
{해당 항목이 있으면 기재}

### 결론
문제 없음. 커밋 가능.
```

---

## 주의사항

- 인자가 없거나 유효하지 않으면 반드시 사용자에게 되물어라. 추측하지 마라.
- 이후 마일스톤 세부 문서가 하나도 없으면 `docs/system-design.md`와 `docs/milestones.md` 정보만으로 검토한다
- 코드 수정은 하지 않는다. 리포트만 출력한다.
