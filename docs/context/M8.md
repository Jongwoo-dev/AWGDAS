## 아키텍처 상태

### 동작하는 것
- TypeScript strict mode + ESM (NodeNext) 빌드 파이프라인
- `npm run typecheck` → 에러 0
- `npm run build` → `dist/` 정상 생성
- `npm test` → 116 tests 전체 PASS (10 test files)
- 공유 타입 정의 (M1): 리터럴 유니온 4개, 에이전트 간 전달 인터페이스 8개, 상태/설정 4개
- 유틸리티 (M2): logger, fileManager, manifest, anthropicClient (3회 재시도 + exponential backoff, streaming 방식)
- 상태 머신 (M3): createRoundState, transition, canRetry, incrementRetry, 데이터 슬롯 setter — 불변 상태 관리
- 에이전트 프롬프트 (M4): PL, Planner, Developer, QA 4종 system prompt + barrel export
- PL 에이전트 (M5): `runPLInit` — RoundSpec 생성
- Planner 에이전트 (M5): `runPlanner` — FeatureBreakdown 생성
- JSON 파서 (M5): `parseJsonResponse` — 직접 파싱 + markdown fence 제거 fallback
- Developer 에이전트 (M6): `runDeveloper` — tool_use 루프, write/read/delete_file, manifest 즉시 갱신
- QA 에이전트 (M7): `runQA` — manifest 무결성 검증 + 파일 내용 수집 + AC 판정
- PL 판단 함수 (M7): `evaluateQAResult` (RELEASE/RETRY/FAIL), `generateFailReport` (FAIL 리포트)
- 전체 파이프라인 루프 (M7): stdin → PL_INIT → PLANNER_DEFINE → DEV_IMPLEMENT → QA_REVIEW → PASS→RELEASE→DONE / REJECT→RETRY→DEV_IMPLEMENT 재진입 / REJECT→FAIL→FAILED
- 에이전트 단위 테스트 (M8): PL 7건, Planner 3건, Developer 7건, QA 5건 — API mock 기반
- 파이프라인 통합 테스트 (M8): 데이터 계약 검증 6건 — 실제 roundStateMachine + 에이전트 mock
